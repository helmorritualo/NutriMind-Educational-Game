HYBRID UI RULES

The Unity UI system is hybrid Canvas plus UI Toolkit.

Existing application scene UI designs must be inspected and reused before creating new UI.

Canvas is allowed for existing app scene designs and already-built UI.

UI Toolkit is allowed for new structured screens, quiz lists, quiz panels, menus, and forms.

Do not rebuild valid finished screens from scratch just to change the UI technology.

A screen may use Canvas, UI Toolkit, or hybrid. Choose the simplest reliable option that matches the existing project.

Do not duplicate business logic between Canvas and UI Toolkit. Both must call shared services, presenters, stores, and DTOs.

If a screen mixes Canvas and UI Toolkit, document which layer owns input, focus, and navigation.

Keep static UI clear. Avoid confusing backgrounds behind important labels, buttons, and panels.

ASSET-FIRST RULE

Before creating anything, inspect:

existing scenes
existing Canvas prefabs
existing UI Toolkit assets
owner-provided references
Assets/_Project/Nutrimind/ThirdParty/
existing project variants
existing scripts and assemblies

Use existing owner-provided assets first, then third-party assets, then project-owned variants, then new/generated assets, then placeholders.

Never destructively edit vendor assets. Create project-owned variants, wrappers, adapters, prefabs, or materials.

CODE ARCHITECTURE

Keep MonoBehaviours thin. Use them to adapt Unity lifecycle, scene references, input, visuals, and events.

Put deterministic quiz/session/provider logic in plain C# classes where practical.

Use shared stores and services for session, profile, settings, subject/term data, quiz availability, quiz detail, quiz session, quiz answers, quiz validation, quiz submission, quiz results, safe errors, and navigation.

Do not put official scoring, answer keys, or server authority inside scene MonoBehaviours.

Do not hardcode quiz content in UI scripts.

Do not hardcode future mission content.

Avoid new global singletons or service locators.

Respect assembly boundaries. Runtime assemblies must not reference Editor assemblies.

Do not edit generated csproj or sln files as authoritative project configuration.

C# STYLE

Follow the repository style first.

Otherwise use PascalCase for types, methods, properties, and events. Use camelCase for locals and parameters. Use _camelCase for private fields. Prefer private SerializeField fields over public mutable fields for Inspector references.

Use guard clauses to reduce nesting. Avoid magic strings and magic numbers. Use constants, enums, DTOs, or configuration where appropriate.

Comment why a non-obvious decision exists, not what obvious code already does.

UNITY LIFECYCLE

Validate required serialized references.

Use Awake for internal initialization, OnEnable and OnDisable for symmetric event/callback registration, and OnDestroy for final cleanup when needed.

Avoid unnecessary Update loops.

Avoid scene-wide Find calls, repeated VisualElement queries, and avoidable allocations in hot paths.

Cancel async work when screens close, scenes unload, sessions end, or newer requests replace older ones.

Do not mutate UnityEngine objects from background threads.
